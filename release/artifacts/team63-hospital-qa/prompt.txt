```text
tests/ui/billing.spec.ts
tests/ui/cross-module.spec.ts

Framework rules:
- Use @playwright/test.
- Read URL and credentials from environment variables.
- Never hard-code secrets.
- Use role-based, label-based or test-id locators where available.
- Avoid brittle generated CSS chains and nth-child selectors.
- Use Page Object Model only where it removes duplication.
- Keep assertions inside test cases.
- Use deterministic test data prefixed T63_20260711_.
- Run tests serially where records share application state.
- Do not enable uncontrolled parallel execution against the shared environment.
- Capture screenshots only on failure.
- Retain trace and video on failure.
- Generate HTML and JUnit reports.
- Save UI reports under reports/ui.
- Associate automated test titles with the corresponding TestCaseID.
- Add cleanup where it is safe.
- Never delete pre-existing records.

First automate only the P0 smoke paths:
1. login
2. register patient
3. book appointment
4. detect overlapping appointment handling
5. cancel appointment and verify slot state
6. admit patient and allocate bed
7. discharge patient and verify bed release
8. create bill and verify payment/dashboard state

Run the smoke tests in headed Chromium.
Fix automation defects without changing the intended business assertions.
Update the traceability matrix with script paths and results.
```

PostQode Web Agent explicitly supports generating Playwright scripts and frameworks from browser actions. ([PostQode Docs][5])

Playwright is the right framework here because one Playwright Test project can execute browser tests and API tests through its built-in `request` fixture, while also producing HTML reports and failure traces. ([Playwright][6])

The likely setup commands proposed by PostQode will resemble:

```powershell
npm init -y
npm install --save-dev @playwright/test dotenv
npx playwright install chromium
npx playwright test
npx playwright show-report
```

Approve them **inside PostQode**, rather than independently building the framework elsewhere. PostQode's normal workflow displays exact terminal commands and file diffs for approval. ([PostQode Docs][7])

# 11. The highest-yield bugs to hunt first

| Priority | Flow                | Failure to seek                                         |
| -------- | ------------------- | ------------------------------------------------------- |
| 1        | Appointment overlap | Same patient or doctor booked into conflicting slots    |
| 2        | Cancellation        | Slot remains blocked after cancellation                 |
| 3        | Discharge           | Bed remains occupied after patient discharge            |
| 4        | Bed allocation      | One bed assigned to two active admissions               |
| 5        | Billing             | Zero, negative or duplicate payment accepted            |
| 6        | Dashboard           | Revenue or occupancy summary disagrees with records     |
| 7        | Lab                 | Report saved without valid patient or doctor            |
| 8        | State transition    | Completed/cancelled/discharged state reversed illegally |
| 9        | Duplicate patient   | Near-identical patient accepted without warning         |
| 10       | Access control      | Hidden page or API accessible through direct route      |

These are not random guesses. They come directly from the competition's stated risk areas and cross-module design. ([ATA Events][1])
